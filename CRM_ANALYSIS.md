# Анализ CRM-системы для мастерской по ремонту 3D-принтеров

## Обзор архитектуры

**Стек**: Python 3 + FastAPI + SQLAlchemy 2.0 + SQLite + Jinja2 + Bootstrap 5.3  
**Тип**: Server-Side Rendered (SSR) приложение

---

## 1. КЛИЕНТЫ (Clients)

### Основные возможности
- **Модель**: `Client` (id, full_name, phone, comment, created_at)
- **CRUD операции**: создание, редактирование, удаление
- **Связи**: один клиент → много заказов (one-to-many)
- **Форматирование телефонов**: автоматическая нормализация к формату +7

### Сильные стороны
✅ Автоматическое форматирование телефонов (8→7, 10→+7)  
✅ Защита от удаления клиентов с заказами (проверка связанных заказов)  
✅ Аудит всех операций (создание, изменение, удаление)  
✅ Сортировка по дате создания (новые сверху)

### Потенциальные улучшения
⚠️ **Отсутствует**:
- Дубликаты клиентов (нет проверки по телефону/имени — можно создать несколько клиентов с одним телефоном)
- История взаимодействий (только заказы, нет заметок, звонков, встреч)
- Теги/категории клиентов (VIP, постоянный, новый)
- Экспорт клиентов в Excel/CSV
- Поиск по телефону в реальном времени (есть только фильтрация на странице)
- Email и другие контакты (только телефон)
- Аватар/фото клиента
- Дата рождения и другие персональные данные

### Критические моменты
- При удалении клиента проверяются только заказы, но не другие сущности
- Нет мягкого удаления (soft delete) — только полное удаление
- Отсутствует валидация формата телефона на уровне модели

## 2. ЗАКАЗЫ (Orders)

### Основные возможности
- **Модель**: `Order` с расширенными полями:
  - Тип заказа (repair, print, warranty)
  - Статусы (in_progress, waiting_parts, ready, closed)
  - Привязка к клиенту и исполнителю
  - Предоплата и оценка стоимости
  - Источник клиента (Авито, Профи.ру, Сарафанка)
  - Гарантия и сроки
- **Составные части**:
  - `OrderItem` — услуги/работы
  - `OrderPart` — использованные запчасти (связь со складом)

### Бизнес-логика
1. **Создание заказа**:
   - Выбор клиента из списка
   - Указание типа принтера, дефекта
   - Назначение исполнителя
   - Установка сроков и предоплаты

2. **Добавление услуг** (`OrderItem`):
   - Добавление работ с ценой
   - Автоматический пересчет общей суммы (`_recalc_total`)
   - Запрет на изменение закрытых заказов

3. **Добавление запчастей** (`OrderPart`):
   - Проверка наличия на складе
   - Автоматическое списание со склада
   - Создание записи `StockMovement`
   - Возврат запчасти при удалении из заказа

4. **Закрытие заказа**:
   - Установка статуса "closed"
   - Фиксация даты закрытия (`closed_at`)
   - Блокировка дальнейших изменений

5. **Batch операции**:
   - Массовое закрытие заказов
   - Массовое удаление

### Сильные стороны
✅ Полный аудит всех действий  
✅ Автоматический пересчет суммы при изменении состава  
✅ Интеграция со складом (автоматическое списание/возврат)  
✅ Гибкая система статусов и типов заказов  
✅ Фильтрация и сортировка по множеству параметров  
✅ Защита от изменения закрытых заказов

### Потенциальные проблемы
⚠️ **Требует внимания**:
- Нет автоматического уведомления клиента о смене статуса (хотя изменение статуса логируется в аудит, клиент не получает уведомление)
- Отсутствует отдельная таблица для истории изменений статуса (есть только в общем аудите, что неудобно для анализа)
- Нет контроля сроков (просроченные заказы не выделяются)
- Предоплата не учитывается в итоговой сумме (только как информационное поле)
- Нет связи с платежами/транзакциями

### Критические моменты
- При удалении заказа запчасти возвращаются на склад, создается новая запись `StockMovement` с типом "in" (это правильно для аудита)
- При удалении закрытого заказа запчасти НЕ возвращаются (правильная бизнес-логика — закрытый заказ уже завершен)
- Нет проверки циклических зависимостей при массовых операциях
- Отсутствует валидация бизнес-правил: можно закрыть заказ без услуг и запчастей (нет проверки в `close_order`)
- Можно удалить заказ в любом статусе (включая закрытый), но при удалении закрытого заказа запчасти не возвращаются

## 3. ТОВАРЫ (Products)

### Основные возможности
- **Модель**: `Product` с полями:
  - Название, артикул (уникальный), цвет
  - Количество на складе
  - Себестоимость, стоимость печати, упаковки
  - Варианты (JSON) — наборы/комплекты
  - Изображение
  - Иерархия (parent_id) — родительские/дочерние товары
- **Движение товаров** (`ProductMovement`):
  - Приход (receive)
  - Расход (supply) — поставка по каналам (Ozon, WB, Розница)
- **Упаковка** (`ProductPackaging`, `PackagingItem`):
  - Связь товаров с упаковочными материалами
  - Расчет стоимости упаковки

### Бизнес-логика
1. **Приход товаров** (`/products/receive`):
   - Batch-операция (множество товаров за раз)
   - Автоматическое создание или обновление товара
   - Поиск по артикулу (если существует — обновляет)
   - Запись в `ProductMovement`

2. **Поставка/расход** (`/products/supply`):
   - Batch-операция (множество товаров за раз)
   - Проверка наличия перед списанием
   - Указание канала сбыта (Ozon, WB, Розница)
   - Поддержка наборов (sets × per_set)
   - Запись в `ProductMovement` с destination

3. **Редактирование**:
   - Изменение себестоимости, стоимости печати/упаковки
   - Сохранение вариантов (JSON)

4. **Отчеты**:
   - Еженедельный экспорт (`/export/products-weekly`)
   - Месячный экспорт (`/export/products-monthly`)

### Сильные стороны
✅ Поддержка иерархии товаров (родительские/дочерние)  
✅ Batch-операции для массового прихода/расхода  
✅ Учет стоимости упаковки и печати  
✅ Система вариантов (наборы)  
✅ Отслеживание движения товаров  
✅ Интеграция с разными каналами сбыта

### Потенциальные проблемы
⚠️ **Требует внимания**:
- Нет автоматического расчета себестоимости на основе компонентов
- Отсутствует резервирование товаров (зарезервировано для заказа) — товары могут быть проданы дважды
- Нет минимального остатка для автозаказа (в отличие от `Part.min_stock`)
- Варианты (JSON) не валидируются на уровне модели (может быть невалидный JSON)
- Нет исторической себестоимости (только текущая, нельзя отследить изменение цены во времени)
- При batch-операциях нет транзакционной защиты (если один товар не прошел, остальные все равно обработаются)

### Критические моменты
- При удалении товара проверяются связанные движения (`ProductMovement`) — если есть движения, товар нельзя удалить (правильная защита)
- Нет блокировки товара при недостаче (только проверка перед поставкой)
- Отсутствует интеграция с внешними маркетплейсами (только учет)
- При удалении товара не проверяются дочерние товары (если есть иерархия) — может привести к orphaned records, так как нет cascade delete для дочерних товаров

## 4. СКЛАД ЗАПЧАСТЕЙ (Warehouse/Parts)

### Основные возможности
- **Модель**: `Part` с полями:
  - Название, артикул (уникальный)
  - Закупочная цена
  - Количество на складе
  - Минимальный остаток (`min_stock`) — для автозаказа
- **Движение запчастей** (`StockMovement`):
  - Приход (receive)
  - Расход (out) — при добавлении в заказ
  - Возврат (in) — при удалении из заказа
  - Связь с заказом (`order_id`)

### Бизнес-логика
1. **Приход запчастей** (`/warehouse/receive`):
   - Batch-операция
   - Автоматическое создание или обновление
   - Поиск по артикулу
   - Обновление закупочной цены
   - Запись в `StockMovement`

2. **Использование в заказах**:
   - Проверка наличия перед добавлением
   - Автоматическое списание при добавлении в заказ
   - Автоматический возврат при удалении из заказа
   - Фиксация цены использования

3. **Редактирование**:
   - Изменение названия, цены, минимального остатка

4. **Удаление**:
   - Проверка использования в заказах
   - Запрет на удаление используемых запчастей

### Сильные стороны
✅ Минимальный остаток для контроля запасов  
✅ Полная интеграция с заказами (автоматическое списание/возврат)  
✅ История всех движений (`StockMovement`)  
✅ Защита от удаления используемых запчастей  
✅ Batch-операции для массового прихода

### Потенциальные проблемы
⚠️ **Требует внимания**:
- Нет автоматического уведомления о низком остатке (хотя есть `min_stock`, но нет проверки и уведомлений)
- Отсутствует автозаказ при достижении `min_stock`
- Нет инвентаризации (сверка фактического наличия с учетным)
- Отсутствует резервирование запчастей для заказов (может возникнуть ситуация, когда запчасть есть на складе, но уже обещана другому заказу)
- Нет учета брака/возвратов поставщикам
- При массовом приходе нет валидации дубликатов (можно дважды добавить одну запчасть)

### Критические моменты
- При удалении запчасти из заказа возвращается цена, зафиксированная в `OrderPart.price` (не текущая цена запчасти), что правильно для аудита
- Нет проверки циклических возвратов (можно бесконечно добавлять/удалять, но это нормальная бизнес-логика)
- Отсутствует валидация артикула (может быть слишком коротким/длинным, нет ограничения длины на уровне модели)
- Нет уникальности названия (только артикул уникален)

## 5. ОБЩИЕ НАБЛЮДЕНИЯ

### Сильные стороны системы
✅ **Целостность данных**:
- Foreign Key constraints на всех связях
- Cascade delete для зависимых сущностей
- Проверки перед удалением (клиенты с заказами, запчасти в заказах)

✅ **Аудит и отслеживание**:
- Все операции логируются через `_audit()`
- Автоматическое создание уведомлений
- История движений склада/товаров

✅ **Бизнес-логика**:
- Автоматический пересчет сумм
- Интеграция склада с заказами
- Гибкая система статусов

✅ **UX/UI**:
- Мобильная адаптация
- Batch-операции для массовых действий
- Фильтрация и сортировка

### Критические проблемы
❌ **Отсутствует**:
1. **Система уведомлений клиентам** — нет SMS/email о смене статуса заказа (критично для бизнеса)
2. **Финансовый модуль** — нет учета платежей, долгов, прибылей/убытков (нельзя отследить финансовое состояние)
3. **Аналитика и отчеты** — нет дашбордов с KPI, графиками продаж, анализом популярных услуг
4. **Резервирование товаров** — нет механизма бронирования (товар может быть продан дважды)
5. **Инвентаризация** — нет сверки фактического наличия с учетным
6. **Защита от дубликатов** — нет проверки по телефону/артикулу при создании
7. **Мягкое удаление** — все удаляется безвозвратно (нельзя восстановить случайно удаленные данные)
8. **Валидация бизнес-правил** — можно закрыть заказ без услуг, удалить заказ в любом статусе
9. **Отдельная история изменений статусов** — изменения логируются в общий аудит, но нет удобной таблицы для анализа переходов между статусами
10. **Контроль сроков** — просроченные заказы не выделяются автоматически

### Рекомендации по улучшению

#### Высокий приоритет (критично для бизнеса)
1. **Система уведомлений клиентам** (SMS/email при смене статуса заказа) — клиенты должны знать о статусе их заказа
2. **Финансовый модуль** (платежи, долги, прибыль) — без этого нельзя вести бизнес
3. **Защита от дубликатов** (проверка по телефону/артикулу) — иначе будет хаос в данных
4. **Мягкое удаление** (soft delete) — возможность восстановить случайно удаленные данные
5. **Валидация бизнес-правил** — запрет на закрытие заказа без услуг, удаление закрытых заказов

#### Средний приоритет (важно для эффективности)
6. **Резервирование товаров** (бронирование для заказов) — предотвратит двойную продажу
7. **Автоматические уведомления** о низком остатке (при достижении `min_stock`)
8. **Инвентаризация** (сверка фактического наличия) — критично для учета
9. **История изменений статусов** (аудит переходов) — для анализа скорости обработки заказов
10. **Контроль сроков** — выделение просроченных заказов, автоматические напоминания

#### Низкий приоритет (улучшение UX)
11. **Экспорт клиентов** в Excel/CSV
12. **Теги/категории** для клиентов (VIP, постоянный)
13. **Автоматический расчет себестоимости** на основе компонентов
14. **Интеграция с маркетплейсами** (Ozon, WB API)
15. **Мобильное приложение** для мастеров (чек-ин/аут, статусы заказов)

## 6. БЕЗОПАСНОСТЬ И ЦЕЛОСТНОСТЬ ДАННЫХ

### Что реализовано хорошо
✅ Foreign Key constraints  
✅ Cascade delete для зависимостей  
✅ Проверки перед удалением  
✅ Аудит всех операций  
✅ Транзакционность (session.commit)

### Что требует улучшения
⚠️ **Отсутствует**:
- Оптимистичная блокировка (optimistic locking) для предотвращения race conditions (два пользователя могут одновременно изменить один заказ)
- Валидация бизнес-правил на уровне модели (например, проверка что `total_price >= 0`, `quantity >= 0`)
- Защита от SQL injection (хотя SQLAlchemy защищает автоматически, но лучше использовать параметризованные запросы везде)
- Rate limiting для API endpoints (защита от DDoS и брутфорса)
- Бэкапы БД (есть backup.sh с SQLite backup API, но нет автоматизации через cron и проверки восстановления)
- Транзакционная целостность при batch-операциях (если один элемент не прошел, все равно коммитятся остальные — это может быть как плюсом, так и минусом в зависимости от бизнес-логики)
- Проверка прав доступа на уровне роутеров (есть система прав через `_has_permission`, но она используется только в шаблонах для отображения UI, на уровне роутеров нет проверки — любой авторизованный пользователь может выполнять любые операции)

## 7. ЗАКЛЮЧЕНИЕ

CRM-система хорошо подходит для базовых задач мастерской по ремонту 3D-принтеров. Реализованы основные функции:
- ✅ Управление клиентами (с форматированием телефонов)
- ✅ Управление заказами с полной интеграцией склада (автоматическое списание/возврат запчастей)
- ✅ Учет товаров и запчастей (с историей движений)
- ✅ Аудит и отслеживание всех операций
- ✅ Мобильная адаптация
- ✅ Batch-операции для массовых действий

**Основные недостатки** (критично для бизнеса):
- ❌ Отсутствие финансового модуля (нельзя отследить прибыли/убытки)
- ❌ Нет системы уведомлений клиентам (клиенты не знают о статусе заказа)
- ❌ Отсутствует аналитика и отчетность (нет KPI и графиков)
- ❌ Нет защиты от дубликатов (хаос в данных)
- ❌ Нет мягкого удаления (нельзя восстановить удаленные данные)
- ❌ Отсутствует резервирование товаров (возможна двойная продажа)

**Рекомендации** (в порядке приоритета):
1. **Высокий приоритет**: Добавить финансовый модуль, систему уведомлений, защиту от дубликатов, мягкое удаление, валидацию бизнес-правил
2. **Средний приоритет**: Резервирование товаров, автоматические уведомления о низком остатке, инвентаризация, история изменений статусов, контроль сроков
3. **Низкий приоритет**: Экспорт клиентов, теги/категории, автоматический расчет себестоимости, интеграция с маркетплейсами

**Вывод**: Система готова к использованию для базовых операций, но требует доработок для полноценного бизнес-процесса. Наиболее критично добавить финансовый модуль и систему уведомлений клиентам — без этого невозможно эффективно вести бизнес.
